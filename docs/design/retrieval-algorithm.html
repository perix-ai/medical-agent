HTML design document
目标读者：team 内部
状态：草稿（待评审）
范围：定义 medical-agent 的索引与检索算法、数据模型、evaluation 框架
1. 目标与非目标
1.1 目标
  • 在 ~10 万份中文医疗文档（指南 / 共识 / 临床路径 / RCT 论文 / 教材章节）上构建可用于 临床诊断辅助 的检索系统。
  • 优化指标顺序：临床安全性 > 引用可追溯性 > 答案准确率 > 召回率 > 延迟 > 成本。
  • 检索结果必须满足：每条进入 LLM 的临床声明都能精确回标到原文段落（含 doc_id / page / bbox / span）。
  • 索引策略层可独立替换、独立评测，便于 ablation。
1.2 非目标
  • 不追求"开放医疗诊断 100% 准确率"——这是问题性质决定的不可达上限。我们追求 受限场景（疾病已知 / 决策树封闭）下错答率 < 1% + 不确定时显式 abstain。
  • 不内化知识：LLM 不重写指南，只在原文段落基础上做组合与引用。
  • 不做病历存储 / EMR 集成；这些在上游 Agent 层处理。
  • 不做实时 streaming 索引；按批处理设计。
1.3 设计原则
  • 原文不动：解析产物保留页码、bbox、原始字符 offset，所有 EvidenceUnit 都能反查原文。
  • 分层解耦：解析 / chunking / 索引 / 检索 / 重排 / 证据聚合 / 回标校验，每层独立模块、独立 metric。
  • 结构优先：先抽出文档的结构化骨架（章节、推荐意见、表格、引用），再做向量化。
  • abstain over hallucination：检索置信度低于阈值时直接拒答，不让 LLM 兜底。

3. 数据模型

所有解析产物落地为 JSONL，便于版本化与 diff。schema 定义在 src/medical_agent/schema/，用 pydantic 强约束。

3.1 Document

一份原始文档（PDF / DOCX / HTML 解析后的根对象）。

class Document(BaseModel):
    doc_id: str                       # 稳定哈希: sha256(source_path + content)[:16]
    source_path: str                  # raw/2022-丙型肝炎防治指南.pdf
    title: str
    authors: list[str] = []
    publisher: str | None             # 中华医学会 / NEJM / ...
    publication_date: date | None
    version: str | None               # "2022年版"
    doc_type: Literal["guideline", "consensus", "rct", "review", "pathway", "textbook"]
    language: str = "zh"
    diseases: list[str] = []          # ICD-10 codes + 中文标签 (后处理填)
    evidence_system: Literal["GRADE", "Oxford", "AHA", "none"] = "none"
    target_population: list[str] = [] # ["成人", "孕妇"] 等
    sections: list[Section]
    tables: list[Table]
    figures: list[Figure]
    recommendations: list[Recommendation]
    references: list[Reference]
    raw_markdown_path: str            # MinerU 输出的 .md 路径
    parse_meta: ParseMeta              # 解析器版本、时间戳、警告
3.2 Section

章节，递归结构。保留 page 范围与字符 span，使任何 EvidenceUnit 都能回到原文。

class Section(BaseModel):
    section_id: str                   # doc_id + path hash
    path: list[str]                   # ["治疗", "DAA 方案", "基因 1b 型"]
    level: int                        # 1, 2, 3...
    title: str
    text: str                         # 该节正文（不含子节），保留 markdown
    page_start: int
    page_end: int
    char_span: tuple[int, int]        # 在 raw_markdown 中的字符 offset
    children: list[Section] = []
    table_refs: list[str] = []        # 引用的 table_id
    figure_refs: list[str] = []
3.3 临床结构化源实体：Recommendation / DiagnosticRule

推荐意见 是解析层源实体，不是检索层级。它从指南原文中抽取，保留推荐原文、证据等级、推荐强度、人群条件、上下文和引用文献。检索时不会直接把 Recommendation 当作层级节点，而是由它生成一个或多个 EvidenceUnit 投影。

class SourceEntity(BaseModel):
    entity_id: str
    doc_id: str
    section_id: str | None
    entity_type: Literal[
        "recommendation",
        "diagnostic_rule",
        "table",
        "figure",
        "reference"
    ]
    page_start: int | None
    page_end: int | None
    span_refs: list[SpanRef] = []


class Recommendation(SourceEntity):
    rec_id: str
    doc_id: str
    section_id: str                   # 所在章节
    number: str | None                # "推荐意见 7"
    statement: str                    # 推荐意见原文 (不改写)
    evidence_grade: str | None        # "A1", "B2", "1A"
    recommendation_strength: str | None  # "强推荐", "弱推荐"
    evidence_system: str | None       # "GRADE" 等
    population: str | None            # "基因 3b 型代偿期肝硬化"
    intervention: str | None          # "索磷布韦/维帕他韦 + 利巴韦林"
    comparator: str | None
    outcome: str | None
    contraindications: list[str] = []
    page: int
    bbox: tuple[float, float, float, float] | None
    surrounding_context: str          # 推荐意见前后各 N 字的论证文本
    cited_refs: list[str] = []        # 引用的文献 ref_id


class DiagnosticRule(SourceEntity):
    rule_id: str
    condition: dict                   # 结构化条件表达式
    conclusion: str
    required_facts: list[str] = []
    exclusion_facts: list[str] = []
    source_quote: str                 # 规则对应的原文

投影关系：

  • Recommendation 和 DiagnosticRule 都是临床结构化源实体。
  • 它们不参与文档层级嵌套；它们通过 section_id 挂到所在章节。
  • 检索时由它们生成 recommendation 或 diagnostic_rule 类型的 EvidenceUnit。
3.4 结构化源实体：Table, Figure, Reference

表格、图片和引文都按独立结构化实体建模。它们不是 EvidenceUnit 的附属字段；EvidenceUnit 只是引用这些源实体生成的检索投影。

  • Table.cells: list[Cell]，每个 cell 带 row/col header（不 flatten 成纯文本）。
  • Figure 至少存 caption + OCR 文本；流程图可后续接 VLM 抽 caption。
  • Reference.ref_id 用于 Recommendation.cited_refs 反向连接。
class Table(SourceEntity):
    table_id: str
    caption: str | None
    cells: list[Cell]
    row_headers: list[str] = []
    col_headers: list[str] = []


class Figure(SourceEntity):
    figure_id: str
    caption: str | None
    image_path: str | None
    ocr_text: str | None


class Reference(SourceEntity):
    ref_id: str
    raw_text: str
    doi: str | None
    title: str | None
    journal: str | None
    year: int | None

投影关系：

  • Table 可以生成完整表格 EvidenceUnit，也可以按行生成多个 table_row EvidenceUnit。
  • Figure 默认作为 display context；诊断流程图、风险分层图等可生成 diagnostic_rule EvidenceUnit。
  • Reference 默认不生成 EvidenceUnit，只参与证据链和引用关系。
3.5 EvidenceUnit

EvidenceUnit 是检索层和引用层的最小证据单元。它可以来自正文段落，也可以来自任意结构化源实体。系统层级仍是三层：Document -> Section -> EvidenceUnit；结构化源实体是 EvidenceUnit 的来源，不是额外层级。

Document
  └── Section
        ├── section_text
        ├── Recommendation / DiagnosticRule / Table / Figure / Reference  # 结构化源实体
        └── EvidenceUnit                                                  # 检索投影
              - paragraph
              - recommendation
              - diagnostic_rule
              - table
              - table_row
              - figure_context
class EvidenceUnit(BaseModel):
    unit_id: str
    doc_id: str
    section_id: str
    unit_type: Literal[
        "paragraph",
        "recommendation",
        "diagnostic_rule",
        "table",
        "table_row",
        "figure_context"
    ]
    source_entity_type: str | None     # recommendation / diagnostic_rule / table / figure / reference
    source_entity_id: str | None       # rec_id / table_id / figure_id / ref_id
    text: str                         # 用于 embedding 与 BM25 的内容
    display_text: str                 # 给 LLM 的内容（可比 text 长，含必要上下文）
    metadata: EvidenceMetadata        # disease, year, evidence_grade, page...
    span_refs: list[SpanRef]          # 回标用的 (doc_id, page, char_span)
    embeddings: dict[str, list[float]] = {}  # 索引时填，runtime 不传

约束：

  • EvidenceUnit 必须能通过 source_entity_id 或 span_refs 反查到源实体/原文。
  • Recommendation、DiagnosticRule、Table、Figure、Reference 是结构化源实体。
  • EvidenceUnit 是检索投影，同一个源实体可以生成 0、1 或多个 EvidenceUnit。
  • doc_summary 与 section_summary 是 Document / Section 的路由投影，不属于 EvidenceUnit 层。
4. 解析层（Parsing）
4.1 主解析器：MinerU
  • 输入：PDF；输出：Markdown + tables(JSON) + figures(PNG + caption)。
  • 选择理由：中文文档 layout / 公式 / 表格综合表现最好；支持 GPU 加速；MIT 协议。
  • 集成方式：subprocess 调用 mineru CLI 而非 import 库，隔离 Python 依赖冲突；解析产物落 data/parsed/<doc_id>/。
4.2 fallback：PyMuPDF + 规则
  • 当 MinerU 解析失败 / 超时时，退化为 PyMuPDF 抽文本 + camelot 抽表格。
  • 规则不追求高质量，只保证 pipeline 不阻塞。失败的文档进入 data/parse_failures.jsonl 待人工处理。
4.3 后处理（structure extraction）

解析后由 parsing/post.py 提取结构化对象：

  1. 章节切分：基于 markdown heading + 中文章节正则（^第[一二三四五六七八九十]+[章节]、^\d+(\.\d+)*\s+）。
  2. 推荐意见抽取：正则 + LLM 判别。规则覆盖 推荐意见\s*\d+、推荐\s*[:：]、Recommendation \d+ 等。证据等级用 \(([A-D]\d?|[1-5][a-c]?)\) 抓取后归一化到统一 enum。
  3. 表格规范化：MinerU 输出的 HTML/JSON 表格 → 我方 Table 模型；保留 row/col header；跨页表格按 caption 合并。
  4. 引用解析：参考文献节用规则切，每条用 pyalex / crossref / regex 抽 (作者, 年份, 期刊, DOI)。
  5. 元数据补全：disease, target_population 等用 LLM 在 doc 摘要 + 标题上抽，归一化到 CMeKG（见 §6）。
4.4 解析质量门

每份文档输出 parse_quality.json：

  • text_coverage：抽出文本字符数 / PDF 总字符数（PyMuPDF 估算）
  • table_count / table_extraction_rate
  • recommendation_count
  • heading_consistency（章节层级是否单调）
  • warnings: list[str]

低于阈值的文档标 quality=low 但仍进索引（带 metadata 标志，检索时降权）。

5. 三层组织与多类型证据单元

这是本期重点验证的策略之一。设计上是 三层组织结构 + 证据单元类型化 + overlap by structure，不按字符数硬切。

5.1 三层组织结构
层级 检索投影 用途 何时召回
Document doc_summary 路由 / 粗筛，包含标题、摘要、病种、版本 第一阶段 routing，决定查哪些文档
Section section_summary 章节路由，定位诊断、治疗、筛查、随访等章节 第二阶段 routing，决定查哪些章节
EvidenceUnit 按 unit_type 索引 原文证据，包含普通段落、表格、推荐意见、诊断规则 最终召回，进入 rerank / LLM / citation verifier
5.2 EvidenceUnit 类型
类型 来源 字符数（中文） 用途
paragraph 章节正文段落 300–800 普通原文证据
table 完整表格 可较长 给 LLM 展示完整决策表
table_row 表格单行 + 表头 80–300 精确命中表格规则
recommendation 指南推荐意见 80–300 强证据，检索时提权
diagnostic_rule 诊断标准 / 风险分层规则 80–300 规则匹配与可解释诊断
figure_context 诊断流程图 / 风险分层图 / 图注 OCR 80–500 图像类临床证据的文本化检索投影
5.3 切分规则
  • Document 投影：title + 自动摘要（LLM 在 sections[0:3] 上生成 ≤300 字摘要，避免随机抽段）。
  • Section 投影：每个 Section 生成 section_summary；长章节仍保留章节对象本身，但不直接把整章作为最终证据。
  • paragraph EvidenceUnit：以 §3.2 的 Section 为边界；若正文过长，按子段落（\n\n + 句号）切，保留 path 前缀作为上下文头（"治疗 > DAA 方案 > 基因 1b 型：……"），相邻 EvidenceUnit 重叠 1 句。
  • recommendation EvidenceUnit：statement + surrounding_context（前 N 句论证 + 后 N 句条件说明），强制单独成 EvidenceUnit。元数据 unit_type=recommendation，检索时可单路提权。
  • diagnostic_rule EvidenceUnit：从诊断标准、风险分层、排除条件、检查阈值中抽取，绑定结构化 condition 与原文 quote。
5.4 display_text vs text
  • text：纯净的检索文本，参与 embedding / BM25。不含 path 前缀以避免噪声。
  • display_text：给 LLM 看的版本，包含 path 前缀 + parent section 摘要 + 表格引用展开。
  • 二者分离让 retrieval 信号干净，LLM 上下文丰富。
5.5 表格 EvidenceUnit（特例）
  • 大表（>10 行）按 row 切：每行 + col header → 一个 table_row EvidenceUnit。
  • 表整体也存一个 table EvidenceUnit，display_text 是完整 markdown 表格。
  • 这样既能精确召回"基因 3b 型 + 代偿期肝硬化 → 索磷布韦/维帕他韦 + 利巴韦林 12 周"这种 row-level 命中，又能在 LLM 端拿到完整决策表。
5.6 反例（明确不做）
  • 不做固定 512 / 1024 token 切分。
  • 不做语义无关的 sliding window。
  • 不让普通段落切分边界穿越 Recommendation 实体。
6. 索引层
6.1 多路索引设计
索引 引擎 内容 用途
BM25 SQLite FTS5 起步，1M 文档前迁 ES evidence_unit.text + 医学分词词典 术语精确匹配（"HCV 基因 3b 型"）
Dense FAISS（IVF-PQ）→ Milvus bge-m3 dense (1024 dim) 语义召回
Sparse 同 BM25 引擎复用，存 bge-m3 sparse 输出 bge-m3 sparse 学习型术语扩展（"HCC↔肝细胞癌"）
Metadata SQLite / Postgres 全部 EvidenceMetadata 字段 必经过滤层
KG（可选 v0.2） NetworkX 起步，量大后 Neo4j 实体-关系图 多跳推理 / disease→guideline 路由
6.2 中文医学分词
  • jieba + 自定义词典（合并 CMeKG / ICD-10 / 药品通用名）
  • 路径：configs/medical_lexicon.txt，每行 术语\t权重\t词性。
  • 同义词归一化（"丙肝 / HCV / 丙型病毒性肝炎" → CUI）由独立的 normalizer 模块处理，不污染 BM25 分词。
6.3 必须索引的元数据字段（检索时必经 filter）
  • publication_year（旧版指南诊断时强制排除或显式标注）
  • doc_type / evidence_system / evidence_grade
  • diseases（CMeKG 归一化 ID + 文本）
  • target_population（成人 / 儿童 / 孕妇 / 老年）
  • version_replaced_by（旧版指针）
6.4 索引构建流程 
parsed Document  ──► unit extraction  ──► 并行：
                                      ├─ embed dense (bge-m3 dense)
                                      ├─ embed sparse (bge-m3 sparse)
                                      ├─ build BM25 doc
                                      └─ extract metadata + entities
                                  ──► 落 4 个索引 + 1 张 evidence_units 主表

主表 evidence_units 是真相源，4 个索引都用 unit_id 做 join key。索引可独立重建。

6.5 实体归一化
  • 工具：spacy + CMeKG 词典 + 自训规则。v0.1 不上深度 NER，规则 + 词典覆盖 80% 即可。
  • 归一化映射存 data/entity_map.parquet，键为 surface form，值为 canonical CUI。
7. 检索流水线

核心原则：先路由，再取证。检索不是直接在全库 EvidenceUnit 上召回，而是按三层候选集逐步收窄：

Query
  └─► Query Analysis
        └─► Doc Routing        -> candidate_doc_ids
              └─► Section Routing   -> candidate_section_ids
                    └─► Evidence Retrieval -> candidate_unit_ids
                          └─► Rerank / Aggregate / Verify / Answer

每一层都只把候选 ID 传给下一层；下一层的检索必须带上上一层的候选集 filter。这样可以避免"问题问丙肝，证据跑到胃癌指南"这类跨文档误召回。

7.1 Query analysis
  • LLM（本地 7B 即可）做三件事：
  1. 口语 → 医学术语（"我得了丙肝" → "HCV / 丙型肝炎"）
  2. 同义词扩展（最多 5 个候选）
  3. 抽出 query 中的实体并归一化（疾病 / 症状 / 药物 / 检查 / 人群条件）
  • 输出：
{
  "normalized_query": "HCV RNA阳性是否可诊断现症HCV感染",
  "expansions": ["丙型肝炎病毒RNA", "HCV感染", "现症感染"],
  "entities": [
    {"type": "disease", "text": "丙型肝炎", "cui": "HCV"},
    {"type": "lab_test", "text": "HCV RNA"}
  ],
  "required_filters": {
    "diseases": ["HCV"],
    "target_population": [],
    "doc_type": ["guideline", "consensus"],
    "publication_year_min": null,
    "prefer_latest": true
  },
  "query_intent": "diagnosis"
}

Filter 比如"孕妇"出现时强制 target_population contains 孕妇；"儿童"出现时强制匹配儿童人群或通用人群；如果 query 中明确指定年份/版本，则关闭 prefer_latest。

7.2 Doc routing：文档级候选集

目标：从 10 万文档中选出少量可能相关的文档，输出 candidate_doc_ids。

输入索引：

  • doc_summary：title + abstract + diseases + doc_type + version + publisher + target_population。
  • 文档元数据表：publication_year, version_replaced_by, doc_type, diseases, target_population, authority_level。

流程：

  1. 应用硬过滤：
  • doc_type 必须在 query 允许范围内；诊断类问题默认优先 guideline / consensus / pathway。
  • 若 query 识别出疾病，diseases 必须命中疾病或其上位类；无法识别疾病时不加 disease 硬过滤。
  • 若 query 识别人群（儿童、孕妇、老年），文档 target_population 必须匹配该人群或标为通用人群。
  • 默认排除 version_replaced_by is not null 的旧版指南；若新版本候选为空，才允许旧版进入并加 warning。
  1. 在过滤后的文档集合上做多路召回：
  • BM25：normalized_query + expansions，top 50。
  • Dense：normalized_query，top 50。
  • Sparse：normalized_query，top 50。
  1. RRF 融合，按 authority_level、publication_year、doc_type 做轻量加权。
  2. 输出 top N 文档，默认 N=8。

排除规则：

  • doc_type=rct/review/textbook 不用于直接诊断结论，除非 query 明确要求研究证据或指南不足。
  • 旧版指南被新版替代时默认排除，不和新版一起融合。
  • 如果 top 文档的 disease 与 query disease 冲突，直接丢弃。

失败条件：

  • candidate_doc_ids 为空 → abstain，原因 no_matching_document。
  • top1 doc score 低于阈值且没有明确疾病实体 → abstain，原因 document_routing_low_confidence。
7.3 Section routing：章节级候选集

目标：在候选文档内定位具体章节，输出 candidate_section_ids。

输入索引：

  • section_summary：section path + section title + 自动摘要 + section intent tags。
  • Section 元数据：doc_id, path, page_start, page_end, section_intent。

流程：

  1. 强制 filter：doc_id in candidate_doc_ids。
  2. 根据 query_intent 加章节意图过滤或提权：
  • diagnosis：优先 诊断 / 实验室检查 / 诊断标准 / 鉴别诊断 / 风险分层。
  • treatment：优先 治疗 / 推荐意见 / 用药 / 禁忌。
  • screening：优先 筛查 / 风险评估 / 早诊早治。
  1. 多路召回：
  • BM25：section title/path/summary，top 50。
  • Dense：section summary，top 50。
  • Sparse：section summary，top 50。
  1. RRF 融合，并按 section intent 与 query intent 匹配度提权。
  2. 输出 top M 章节，默认 M=20，并保留每个章节的 parent/child 邻接信息。

排除规则：

  • query 是诊断判断时，参考文献 / 致谢 / 方法学 / 背景流行病学 章节默认排除，除非没有其他章节命中。
  • 章节所在文档若已被 doc routing 标记为旧版降权，section 继承该 warning。

失败条件：

  • candidate_section_ids 为空 → 回退到 candidate docs 内全文 EvidenceUnit 检索一次；仍为空则 abstain，原因 no_matching_section。
  • top section 与 query_intent 严重不匹配，例如诊断问题只命中治疗章节 → 标 section_intent_mismatch，进入 evidence 阶段但提高 abstain 阈值。
7.4 Evidence retrieval：证据单元召回

目标：在候选章节内召回可引用的最小证据单元，输出 candidate_unit_ids。

输入索引：

  • EvidenceUnit 主表与索引：paragraph / recommendation / diagnostic_rule / table / table_row / figure_context。
  • 元数据：doc_id, section_id, unit_type, page, diseases, target_population, evidence_grade, sensitivity_level。

流程：

  1. 强制 filter：
  • section_id in candidate_section_ids。
  • 若 query 有明确疾病/人群，EvidenceUnit 的 metadata 必须命中或继承自上级 Section/Document。
  • 默认排除 quality=low 的解析产物；若无结果再放开并加 warning。
  1. 按 query intent 选择召回路：
  • diagnosis：diagnostic_rule 与 recommendation 提权，paragraph/table_row 参与补充。
  • screening/risk：diagnostic_rule/table_row/recommendation 提权。
  • definition/background：paragraph 提权。
  1. 多路召回：
  • BM25：normalized_query + expansions，top 80。
  • Dense：normalized_query，top 80。
  • Sparse：normalized_query，top 80。
  • Rule lookup：若 query facts 能匹配 diagnostic_rule.condition.required_facts，直接加入候选。
  1. RRF 融合，并按 unit_type 加权：
  • diagnostic_rule: +强提权
  • recommendation: +提权
  • table_row: +中等提权
  • paragraph: 默认
  • table: 不直接作为 top evidence，除非其 row 命中或需要完整上下文。
  1. 为命中的 EvidenceUnit 自动补充邻接上下文：
  • paragraph 命中时，可带前后 1 个 paragraph 作为 display_context，但证据 ID 仍只指向原命中 unit。
  • table_row 命中时，额外带完整 table 的 display_text。
  • recommendation 命中时，带 surrounding_context。

排除规则：

  • EvidenceUnit 没有 span_refs 或无法回标页码时，不允许进入最终 evidence。
  • 与 query 人群冲突的 EvidenceUnit 直接排除，例如儿童问题命中成人专用推荐。
  • diagnostic_rule 的 required facts 与病例事实明显矛盾时，不作为支持证据，但可作为"不满足项"返回。

失败条件：

  • candidate_unit_ids 为空 → abstain，原因 no_matching_evidence。
  • 命中结果都没有可回标 span → abstain，原因 no_traceable_evidence。
7.5 Reranker
  • bge-reranker-v2-m3，cross-encoder。
  • 输入 (query, evidence_unit.text)，输出 score。
  • 截断到 top 8（默认值，可调）。
  • 若 reranker top1 score < threshold（待标定），整个 query 标 low_confidence=True。
  • reranker 只在 candidate_unit_ids 内排序，不允许引入上一层未通过的文档或章节。
7.6 Evidence Aggregation
  • 同一 Recommendation 的多个 EvidenceUnit 命中 → 合并为单个证据条目。
  • 不同文档对同一 (population, intervention) 的推荐若 结论冲突，保留两个独立证据条目并标 conflicting=True，绝不自动合并。
  • 输出：list[Evidence]，每个 Evidence 含 statement, grade, sources（多个 EvidenceUnit）, conflict_with。
7.7 Answer composition
  • LLM prompt 强制 schema 输出，每条 claim 必须带 evidence_ids: list[unit_id]。
  • prompt 模板：
  • 系统 prompt 限制不得引入 evidence 之外的临床事实
  • few-shot 示例展示如何说"证据不足，建议……"
7.8 Citation Verifier（回标校验）
  • 对 LLM 输出的每个 claim：
  1. 取声明文本 → embedding
  2. 与该 claim 引用的 evidence 文本算 cosine sim
  3. 若 sim < 0.55（待标定）或 evidence_ids 为空 → 丢弃此 claim
  • 丢弃后若答案为空，整体回退到 abstain + 列出 top evidence 让用户自决。
  • 这是临床安全的最后一道闸。
7.9 Abstain 策略

触发任一条件即 abstain：

  • query 改写 LLM 拒识 / 超出医学范畴
  • Doc / Section / Evidence 任一层候选集为空且回退失败
  • 全部 evidence 召回路 top1 score 低于阈值
  • 检索结果证据系统冲突（指南互相矛盾）
  • 引用回标全部失败

abstain 时返回结构化 reason，前端可决定是 escalate 给医生还是要求用户补全信息。

8. Evidence Chain 协议
8.1 SpanRef

每个 EvidenceUnit 都携带 span_refs: list[SpanRef]：

class SpanRef(BaseModel):
    doc_id: str
    page: int
    char_span: tuple[int, int]        # 在 raw_markdown 中的 offset
    bbox: tuple[float, float, float, float] | None  # 在 PDF 中的位置
    quote: str                        # 实际文本（用于回标校验冗余存）
8.2 引用 ID 规范
  • 对外暴露的 citation id 格式：{doc_id}#sec={section_id}#rec={rec_id?}#p={page}
  • 解析端到端可逆：从 citation id 能反查到原 PDF 页面 + bbox。
8.3 输出契约

最终回答必须满足：

{
  "answer": "...",
  "claims": [
    {"text": "...", "evidence_ids": ["...", "..."], "confidence": 0.92}
  ],
  "evidences": [
    {"id": "...", "doc_title": "...", "doc_year": 2022, "quote": "...", "page": 12, "bbox": [...]}
  ],
  "abstain": false,
  "abstain_reason": null,
  "warnings": ["旧版指南 (2019) 已被排除"]
}

前端不允许显示没有 evidences 的 claim。

9. 评测框架
9.1 评测集分层
  • L0 自动构造：从指南推荐意见反推 QA（已知答案 → 生成问题），用于回归测试。不作为发布门。
  • L1 临床场景集：医生标注，~200 条，覆盖：
  • 单一指南内决策（"基因 1b 型初治非肝硬化首选方案"）
  • 跨指南检索（"妊娠合并丙肝是否能 DAA 治疗"）
  • 冲突识别（同一问题两份指南推荐不同）
  • abstain 触发（信息不足、超范围）
  • L2 红队集：医生 + 工程师设计的 adversarial 集，专门测幻觉、跨年份混淆、人群错配。
9.2 指标
  • 检索侧：Recall@k（k=5/10/20），nDCG@10，引用覆盖率（answer 中所有 claim 都有 evidence 的比例）。
  • 答案侧：claim accuracy（医生评，每条 claim 对/错/不足以判断），abstain precision / recall，conflict detection F1。
  • 安全侧：错答率（错答 = 有 claim 但与公认指南相反），临床危险错答数（独立计数）。
  • 工程侧：p50 / p95 延迟，索引存储 / 内存占用。
9.3 Ablation 维度

每条配置都用同一份 L1 集跑：

  1. evidence unit extraction：单粒度 section vs 多类型 EvidenceUnit（默认）
  2. 检索路：仅 BM25 / 仅 dense / hybrid（默认）
  3. reranker on/off
  4. citation verifier on/off
  5. recommendation EvidenceUnit 提权 on/off
  6. metadata filter 严格度

每次提交都打 eval_report.json，CI 上回归对比。

9.4 标定阈值
  • reranker low_confidence threshold
  • citation verifier sim threshold
  • abstain 触发组合

这些都不应硬编码，统一放 configs/thresholds.yaml，每次发布前在 L1 上重新标定。

11. 实施路线图
阶段 范围 出口标准
M0：脚手架 目录、pyproject、schema、CLI 占位 python -m medical_agent.cli.ingest --help 可跑
M1：解析跑通 MinerU 集成 + post-processing + parse_quality raw/ 下两份 PDF 完整解析为 data/parsed/；丙肝指南推荐意见数 ≥ 实际数的 90%
M2：证据单元抽取 §5 落地 + 单测 EvidenceUnit 数 / 长度分布 / 推荐意见命中率上 dashboard
M3：单路检索 baseline 仅 BM25，单粒度 section 在 L0 自动 QA 上 Recall@10 报告
M4：混合检索 + 多粒度 dense / sparse / RRF / 多粒度合并 在 L0 上 Recall@10 ≥ M3 + 10pp
M5：reranker + evidence chain citation verifier + abstain L1 集上引用覆盖率 ≥ 95%，错答率 < 5%
M6：评测闭环 + ablation eval CLI + 报告生成 每次 PR 跑 ablation，自动评论变化
M7：扩量 加 1k / 10k 文档 索引大小、构建时间、p95 延迟符合预算

每阶段独立 PR，独立 metric，独立可回滚。

12. 已识别的风险与待决定事项
风险
  1. MinerU 在某些指南上可能 OOM 或表格识别退化：mitigation 是 fallback 链 + parse_failures 队列。
  2. bge-m3 在医学领域的 zero-shot 表现未知：M3 后必须在自建集上量化对比 BGE-M3 / Conan-embedding / PubMedBERT-zh。结果决定是否需要 LoRA。
  3. 临床冲突检测做不准会反向降低体验：M5 之前先做"标识冲突，不自动合并"，不做"判定哪个对"。
  4. Citation verifier 的 sim 阈值与 LLM 偏好高度相关：每次换 LLM 必须重标。
  5. 数据合规：原始指南版权问题。M1 之前确认是否仅限内部使用。
待决定（需要业务输入）
  • [ ] 检索时默认是否按"最新版优先"还是"全部召回 + 警示"？
  • [ ] abstain 后端到端 UX：直接返回 vs 转人工？
  • [ ] 是否需要支持非中文文献？（FDA / NEJM 等）
  • [ ] LLM 选型：本地 vLLM (Qwen2.5-72B / DeepSeek-V3) vs API。涉及成本与合规。
  • [ ] 是否给每条 EvidenceUnit 标"诊断敏感度"（用药剂量、禁忌、急救方案标 HIGH，背景介绍标 LOW）？检索时高敏度 EvidenceUnit 的回标更严格。
